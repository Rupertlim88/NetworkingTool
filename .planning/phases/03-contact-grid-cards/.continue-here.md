<blockers>
- **GSAP stacking context constraint**: Pseudo-elements cannot layer behind GSAP-transformed elements. Must use box-shadow or sibling DOM elements. Permanent constraint.
- **Dark-on-dark contrast**: Shadows on teal/midnight backgrounds need extreme opacity or alternative approaches (white/surface brightness). Standard black shadow values are invisible.
- **Box-shadow layer ordering**: Multiple shadows on same element render first-on-top. A light glow as the last layer gets buried under dark layers above it. Use separate elements for independent shadow rendering.
</blockers>

<context>
The user has a strong vision for premium, 3D-feeling cards. We've exhausted several shadow-based approaches for dark backgrounds and reached the conclusion that traditional dark UIs (Apple's approach) use surface brightness differentiation rather than shadows. The card already has some of this (lighter surface than background, top edge highlight), but the user wants more pronounced depth.

The conversation was at a decision point: whether to abandon shadow-based depth entirely and strengthen surface contrast instead, or continue trying shadow approaches. The user's reference point is the Aurora theme (bright gradient background) where black shadows work naturally.

Key technical learnings this session:
1. Box-shadow layers on same element: first layer renders on top, last on bottom — glow layers get buried
2. Separate elements (wrapper vs inner) render shadows independently — useful for combining glow + dark shadows
3. Same-color-family shadows (cyan glow on teal bg) are invisible regardless of opacity

Current shadow state in CSS:
- Teal: 4 white layers at 3-8% opacity (too subtle)
- Midnight: 4 white layers at 3-8% opacity (too strong)
- Aurora: 4 black layers at original values (looks good)

The hover glow, click animation, and remaining data population are still pending but secondary to resolving the base card depth feel.
